## 网络层安全概述

IP 协议本身没有安全机制：分组头部明文传输、不验证来源、不保证内容未被篡改。任何在转发路径上的节点都能窃听、伪造或重放 IP 分组。要补上这层安全，可以在不同协议层做，IPsec 选择在**网络层**做。

### 为什么在网络层做安全

在网络层加密/认证，意味着安全处理发生在 IP 这一层，对其上的传输层与应用层完全透明：

- **对上层透明**：TCP/UDP 段和应用数据无需任何改动，原本跑在 IP 上的所有流量（HTTP、DNS、SMTP、自研协议……）都自动获得保护。相比之下 TLS 只保护用了 TLS 的那条连接，PGP 只保护邮件。
- **保护所有 IP 流量**：一条 IPsec 通道一旦建立，经过它的全部分组都被统一加密/认证，包括那些自身没有任何安全能力的老协议。
- **可在网关集中部署**：把 IPsec 放在网关（边界路由器/防火墙）上，内网主机不感知，就能让整个站点的对外流量走加密隧道——这正是站点到站点 VPN 的基础。

代价是：网络层看不到应用语义，无法做"按用户/按内容"的细粒度策略；且加密后中间盒（NAT、DPI，见 [4.5](4.5网络层：中间盒技术.md)）难以处理流量。

### 各层安全机制对比

安全可以加在任意一层，层次越低保护越通用、越靠近应用则越能利用应用语义。常见对照：

| 层次 | 代表机制 | 保护范围 | 保护粒度 | 对上层透明 | 典型部署位置 |
|------|---------|---------|---------|-----------|------------|
| 链路层 | WPA2、MACsec | 单段链路（逐链路） | 整个帧 | 透明 | 相邻两节点之间 |
| 网络层 | **IPsec** | 端到端 / 网关到网关 | 整个 IP 分组或其载荷 | 透明 | 主机或网关 |
| 传输层 | TLS | 一条 TCP 连接（进程到进程） | 应用数据流 | 应用需调用 | 端系统进程 |
| 应用层 | PGP、S/MIME | 单个应用对象（如一封邮件） | 应用数据单元 | 不透明 | 应用程序内 |

注：链路层安全只覆盖一跳，分组进入下一段链路要重新加解密，中间路由器能看到明文；IPsec 则可让分组在整条路径上保持受保护。

易混：**IPsec vs TLS**。两者都能做端到端机密性+完整性，但层次不同：TLS 在传输层之上、保护单条连接，应用要主动用它，加密后端口/IP 仍可见；IPsec 在网络层、对应用透明地保护所有流量，还能在网关上集中部署成 VPN。TLS 更适合"某个应用要安全"，IPsec 更适合"整条网络通道要安全"。

## VPN 概述

> **虚拟专用网 (VPN, Virtual Private Network)**
>
> 在公共网络（通常是 Internet）之上，通过隧道与加密技术构建出的、逻辑上专用的网络。它让地理上分散的站点或远程用户"看起来"像连在同一个专用网内，而无需真正铺设专线。

专线（如租用的物理链路）安全但昂贵；公网便宜但不安全。VPN 取两者之长：用公网做承载，用加密/认证把它"伪装"成专线。

### VPN 的两种基本形态

```
站点到站点 (site-to-site)：网关 ↔ 网关，连接两个网络
   总部内网 ──[网关A]══════隧道══════[网关B]── 分部内网
              （公网 Internet）
   内网主机互不感知，加解密由网关完成

远程接入 (remote-access)：客户端 ↔ 网关，单个用户接入内网
   出差笔记本 ─[VPN客户端]═══隧道═══[网关]── 公司内网
              （公网 Internet）
   客户端软件负责加解密，用户像在内网一样访问资源
```

- **站点到站点**：连接两个固定网络（如总部与分支机构），由两端网关建立长期隧道，内网主机无感知。
- **远程接入**：连接单个移动用户与内网，用户机上装 VPN 客户端，临时拨建隧道。

### 隧道技术：封装与解封装

VPN 的核心机制是**隧道 (tunneling)**：把要传输的原始分组当作"数据"，外面再套一层新的分组头，使其能在公网上被正常路由；到达对端再剥掉外层、还原原始分组。

```
封装 (encapsulation)              解封装 (decapsulation)
┌───────────────┐                ┌───────────────┐
│ [新头部][原始分组] │ ──公网路由──► │ 剥掉新头部 → 原始分组 │
└───────────────┘                └───────────────┘
   入口节点添加                       出口节点还原
```

封装让"专网地址"的分组能借公网传输：内层可以是私有地址、甚至是另一种协议，外层用公网可路由地址承载。若再对内层分组加密，公网上的人就只看到外层头和密文，看不到真实通信内容——这就是 IPsec 隧道模式做的事（见下文）。

注：隧道本身只解决"封装传输"，不等于安全。GRE 等纯隧道协议只封装不加密；要安全须叠加加密/认证，IPsec 正是把"隧道 + 加密认证"合在一起。

### 与 MPLS-VPN 的关系与区别

VPN 不止 IPsec 一种实现。运营商常用的 **MPLS-VPN** 工作在多协议标签交换之上（详见 [6.5 MPLS-VPN](6.5链路层：链路虚拟化.md#mpls-vpn)），与 IPsec-VPN 的实现思路完全不同：

| 维度 | IPsec-VPN | MPLS-VPN |
|------|-----------|----------|
| 实现层 | 网络层（IP 之上加安全） | 运营商骨干网（标签转发） |
| 安全手段 | **加密 + 认证**（密码学保护） | **流量隔离**（不同 VPN 标签隔离，默认不加密） |
| 承载网络 | 公共 Internet | 运营商可信骨干网 |
| 信任前提 | 不信任承载网，自己加密 | 信任运营商，靠标签隔离 |
| 谁来运维 | 企业自建自管 | 运营商提供服务 |

一句话区别：**IPsec-VPN 靠"加密"在不可信公网上获得机密性；MPLS-VPN 靠"隔离"在可信运营商网内获得专用性**。前者哪怕分组被截获也无法解读，后者则假设骨干网本身可信、只需把不同客户的流量分开。两者也可叠加（如在 MPLS 专线上再跑 IPsec 加密）。

## IPsec 体系结构

IPsec (IP Security) 不是单一协议，而是 IETF 定义的一套网络层安全**框架**，由三大部分组成：

```
IPsec
├─ AH  (认证头)         提供完整性、数据源认证、抗重放（不加密）
├─ ESP (封装安全载荷)    提供机密性（加密）+ 可选完整性/认证
└─ IKE (密钥交换协议)    自动协商安全参数、建立密钥
```

AH 和 ESP 负责"保护分组"，IKE 负责"协商怎么保护"（密钥、算法等）。三者协作前，通信双方必须先就一套安全参数达成一致，这套参数的载体就是安全关联。

### 安全关联 SA

> **安全关联 (SA, Security Association)**
>
> 通信双方关于"如何保护一个方向的流量"的一组约定，包括所用协议（AH/ESP）、加密与认证算法、密钥、密钥生存期等。SA 是 IPsec 一切保护操作的基础。

SA 有两个关键性质，是考点：

- **SA 是单向的**：一个 SA 只保护一个方向的流量。要双向通信就需要**两个 SA**（A→B 一个，B→A 一个）。
- **SA 由三元组唯一标识**：`(SPI, 目的IP地址, 安全协议)`。
  - **SPI (Security Parameter Index, 安全参数索引)**：32 位，由接收方分配，写在 AH/ESP 头里。收到分组时，接收方靠 SPI（配合目的地址、协议）从本地数据库找到对应 SA，从而知道用哪个密钥、哪种算法来处理。
  - **目的 IP 地址**：SA 终点的地址。
  - **安全协议**：标明这是 AH 还是 ESP 的 SA。

注：因为 SA 单向，"用 ESP 保护 A 与 B 间的双向通信"实际要建立 2 个 SA；若同时用 AH+ESP，则方向数还要再乘协议数。

### SAD 与 SPD

IPsec 用两张数据库管理 SA 和策略：

- **SAD (Security Association Database, 安全关联数据库)**：存放所有**已建立的 SA**。处理分组时，按 `(SPI, 目的IP, 协议)` 在 SAD 中查到对应条目，取出密钥与算法执行加解密/校验。回答的是"用什么参数处理这个分组"。
- **SPD (Security Policy Database, 安全策略数据库)**：存放**安全策略**，规定哪些流量需要保护、怎么保护。对每个分组按选择符（源/目的 IP、端口、协议等）匹配策略，得出三种动作之一：**丢弃 (DISCARD)**、**绕过 IPsec (BYPASS)**、**应用 IPsec (PROTECT)**。回答的是"这个分组要不要、要用哪种保护"。

```
出向分组 ──► 查 SPD（要不要保护？怎么保护？）
                │PROTECT
                ▼
            查/建 SAD（取出 SA：密钥、算法）──► AH/ESP 处理 ──► 发出

入向分组 ──► 按 SPI 查 SAD（取 SA 校验/解密）──► 查 SPD 核对策略 ──► 交付
```

### 传输模式 vs 隧道模式

IPsec 有两种工作模式，区别在于**保护的范围**和**是否新增 IP 头**。

**传输模式 (transport mode)**：只保护 IP 分组的**载荷**（传输层数据），保留原 IP 头。适用于两台**主机之间**的端到端保护。

```
原始分组：
┌────────┬──────────────────────┐
│ IP头    │ TCP/UDP + 数据          │
└────────┴──────────────────────┘

传输模式（以 ESP 为例）：
┌────────┬────────┬──────────────────────┬─────────┐
│ 原IP头  │ ESP头   │ [ TCP/UDP + 数据 ]加密  │ ESP尾/认证 │
└────────┴────────┴──────────────────────┴─────────┘
         └──────── 保护范围：仅载荷 ────────┘
原 IP 头保持明文，源/目的地址对外可见
```

**隧道模式 (tunnel mode)**：把**整个原始 IP 分组**（含原 IP 头）当作载荷保护，外面再加**一个新的 IP 头**。适用于**网关到网关**或主机到网关，是 VPN 的常用模式。

```
隧道模式（以 ESP 为例）：
┌────────┬────────┬───────────────────────────┬─────────┐
│ 新IP头  │ ESP头   │ [ 原IP头 + TCP/UDP + 数据 ]加密 │ ESP尾/认证 │
└────────┴────────┴───────────────────────────┴─────────┘
                  └──── 保护范围：整个原始 IP 分组 ────┘
新 IP 头用网关的公网地址，原 IP 头（含真实内网地址）被加密隐藏
```

两种模式对比：

| 维度 | 传输模式 | 隧道模式 |
|------|---------|---------|
| 保护对象 | 仅原始分组的载荷 | 整个原始 IP 分组 |
| 是否加新 IP 头 | 否，沿用原 IP 头 | 是，加新 IP 头 |
| 原 IP 头是否暴露 | 暴露（地址明文可见） | 隐藏（被封装进密文） |
| 典型场景 | 主机到主机（端到端） | 网关到网关（VPN） |
| 开销 | 小 | 大（多一个 IP 头） |

注：网关代表整个内网收发流量，必须用隧道模式——既要承载内网主机的原始分组（连同其私有地址原 IP 头一起封装），又要用网关自己的公网地址做新 IP 头才能在公网路由。传输模式不加新头，无法承担网关角色。

## AH 与 ESP

AH 和 ESP 是 IPsec 真正"动手保护分组"的两个协议。最关键的区别一句话：**AH 不加密，ESP 才加密**。

### 认证头 AH

> **认证头 (AH, Authentication Header)**
>
> 为 IP 分组提供**完整性、数据源认证、抗重放**，但**不提供机密性（不加密）**。

AH 对分组（包括 IP 头中传输途中不变的字段）计算一个带密钥的校验值 (ICV)，接收方重算比对，从而确认：分组没被篡改（完整性）、确实来自声称的发送方（数据源认证）、不是被重放的旧分组（抗重放，靠序号实现）。

AH 报文主要字段：

```
┌──────────────┬──────────────┬──────────────────┐
│ 下一头部       │ 载荷长度      │ 保留              │
├──────────────┴──────────────┴──────────────────┤
│ 安全参数索引 SPI（定位 SA）                        │
├─────────────────────────────────────────────────┤
│ 序列号 Seq（抗重放，单调递增）                      │
├─────────────────────────────────────────────────┤
│ 完整性校验值 ICV（带密钥的校验，认证数据）           │
└─────────────────────────────────────────────────┘
```

注：AH 把 IP 头中的不变字段也纳入认证，因此能检测对源地址等头部字段的篡改；但也正因如此，它**与 NAT 不兼容**——NAT 改写 IP 头会让 ICV 校验失败。这是实际中 AH 少用的重要原因。

### 封装安全载荷 ESP

> **封装安全载荷 (ESP, Encapsulating Security Payload)**
>
> 提供**机密性（加密）**，并可选地提供完整性、数据源认证与抗重放。是实际部署中绝大多数场景的选择。

ESP 对载荷加密保证机密性，并可同时用 ICV 提供完整性/认证（即"加密 + 认证"一起做）。它的认证范围**不包含外层 IP 头**，所以对 NAT 友好（配合 NAT-T 可穿越 NAT）。

ESP 报文主要字段（载荷夹在头和尾之间）：

```
┌─────────────────────────────────────────────────┐
│ 安全参数索引 SPI（定位 SA）                        │  ┐
├─────────────────────────────────────────────────┤  │ ESP头（明文）
│ 序列号 Seq（抗重放）                               │  ┘
├─────────────────────────────────────────────────┤
│ 载荷数据（被加密：传输模式=传输层数据；隧道模式=原IP包）│  ← 加密范围
├──────────────┬──────────────┬───────────────────┤
│ 填充 Padding  │ 填充长度      │ 下一头部           │  ← ESP尾，也在加密范围
├──────────────┴──────────────┴───────────────────┤
│ 完整性校验值 ICV（可选，对 ESP头+载荷+ESP尾 认证）   │
└─────────────────────────────────────────────────┘
   认证范围：ESP头 + 加密载荷 + ESP尾（不含外层IP头）
```

### AH vs ESP：对比与选用

| 安全服务 | AH | ESP |
|---------|-----|-----|
| 数据完整性 | 是 | 可选（一般启用） |
| 数据源认证 | 是 | 可选（一般启用） |
| 抗重放 | 是 | 可选 |
| **机密性（加密）** | **否** | **是** |
| 认证是否含外层 IP 头 | 含 | 不含 |
| 与 NAT 兼容 | 否 | 是（配合 NAT-T） |

**选用结论：绝大多数场景用 ESP**。原因：

- ESP 本身就能同时提供机密性 + 完整性 + 认证，覆盖了 AH 的能力还多了加密；
- AH 不加密，单独用它内容仍是明文，安全价值有限；
- AH 与 NAT 不兼容，而现实网络普遍有 NAT。

易混：**AH 提供完整性≠提供机密性**。AH 能让你确信"分组没被改、来自真发送方、不是重放"，但**内容仍是明文**，任何人都能看。要"看不见内容"必须用 ESP 加密。AH 唯一相对 ESP 的"优势"是认证范围覆盖外层 IP 头，但这恰恰让它无法穿越 NAT，得不偿失。

### 两种模式下的保护范围对比

把 AH/ESP 与传输/隧道模式组合，保护范围如下：

```
传输模式：
  AH  →  认证 [原IP头(不变字段) + 传输层数据]，不加密
  ESP →  加密/认证 [传输层数据]，原IP头明文暴露

隧道模式：
  AH  →  认证 [新IP头(不变字段) + 整个原IP包]，不加密
  ESP →  加密/认证 [整个原IP包]，新IP头明文、原IP头被隐藏
```

实战最常见的组合是 **隧道模式 + ESP**：整个原始分组（含真实地址）被加密封进新分组，公网上只看到两端网关地址和一堆密文——这正是站点到站点 VPN 的标准做法。

## 密钥管理 IKE

AH/ESP 要工作，必须先有 SA（密钥、算法等参数）。这些参数从哪来？两条路：手工配置，或自动协商。

### 手工 vs 自动密钥管理

- **手工密钥**：管理员在两端手动配好相同的密钥和算法。简单、无需协商协议，但密钥固定不变、难以更新、规模一大就无法维护，仅适合极小型或测试环境。
- **自动密钥管理（IKE）**：由协议动态协商算法、交换密钥、并定期更新。可扩展、支持密钥轮换与前向保密，是实际部署的标准方式。

### IKE / ISAKMP 的两阶段协商

> **IKE (Internet Key Exchange)** 是 IPsec 的自动密钥交换协议，运行在 ISAKMP（定义协商框架与报文格式）之上，负责为通信双方安全地协商出 IPsec 所需的 SA 与密钥。

IKE 采用**两阶段**设计：先建立一条受保护的协商通道，再在这条通道里安全地协商真正用于数据保护的 SA。

```
阶段一 (Phase 1)：建立 IKE SA（也叫 ISAKMP SA）
  目的：双方相互认证 + 用 Diffie-Hellman 协商出共享密钥，
        建立一条双向、受保护的"协商专用"安全通道。
  ┌──────────┐   协商算法 / DH 交换 / 身份认证   ┌──────────┐
  │  发起方   │ ◄──────────────────────────► │  响应方   │
  └──────────┘        建立 IKE SA              └──────────┘

              （此后所有协商都在 IKE SA 保护下进行）
                              │
                              ▼
阶段二 (Phase 2)：建立 IPsec SA
  目的：在阶段一的安全通道里，协商真正用于保护用户数据的
        AH/ESP 的 SA（含密钥、算法、模式）。
  ┌──────────┐   协商 ESP/AH 参数、刷新密钥     ┌──────────┐
  │  发起方   │ ◄──────────────────────────► │  响应方   │
  └──────────┘   建立成对 IPsec SA（双向各一）   └──────────┘
                              │
                              ▼
              数据流量用协商好的 IPsec SA 加密/认证传输
```

为什么分两阶段：阶段一较慢（含 DH 计算与认证）但只需偶尔做一次，建好的 IKE SA 可被多次复用；阶段二在已加密的通道里快速协商、可频繁刷新数据密钥。这样既安全又高效。

注：阶段二协商出的 IPsec SA 同样是**单向成对**的——双向通信需要两个 IPsec SA，与前面 SA 单向性一致。

### 与 Diffie-Hellman 的衔接

IKE 阶段一用 **Diffie-Hellman 密钥交换**让双方在不可信信道上各自算出同一个共享密钥，且窃听者即使看到全部交换报文也算不出该密钥（依赖离散对数难题）。这是 IKE 能"凭空"安全建密钥的数学基础。DH 的公共参数、私钥、共享密钥推导，以及为前向保密而用的 ECDHE，详见 [2.8 密钥交换](2.8应用层：密码学基础.md#密钥交换)，此处不展开。

## 典型部署与例题

### 站点到站点 VPN 的部署思路（概念性）

目标：让总部与分部两个内网经公网安全互通，内网主机无感知。标准方案是 **隧道模式 + ESP**：

```
总部内网                                          分部内网
10.1.0.0/16                                       10.2.0.0/16
   │                                                  │
[网关A]══════════════隧道（公网）══════════════[网关B]
公网 203.0.113.1                              公网 198.51.100.1

数据流（内网主机 10.1.0.5 → 10.2.0.8）：
1) 主机发出原始分组：[IP 10.1.0.5→10.2.0.8 | 数据]
2) 网关A 按 SPD 判定需保护，用 ESP 隧道模式加密整个原分组，
   加新 IP 头：[IP 203.0.113.1→198.51.100.1 | ESP | 密文]
3) 公网只看到两网关地址 + 密文，按新 IP 头正常路由
4) 网关B 按 SPI 查 SAD 找到 SA，解密、剥外层，还原原始分组
5) 还原后的 [IP 10.1.0.5→10.2.0.8 | 数据] 在分部内网投递
```

配置要点（概念层面，不涉及具体厂商命令）：

- 在两端网关定义 SPD：匹配"总部网段 ↔ 分部网段"的流量执行 PROTECT，其余 BYPASS；
- 选择 ESP + 隧道模式，约定加密与认证算法；
- 用 IKE 自动协商：阶段一建 IKE SA（DH + 双方认证），阶段二建成对 IPsec SA；
- 双向通信需要两个 IPsec SA，由 IKE 一并协商。

### 例题一：选传输模式还是隧道模式

> 场景 A：公司两台**服务器**之间要做端到端加密，两台机器都装了 IPsec，且不经过会改地址的设备。
> 场景 B：要在两个分支机构的**边界网关**之间建 VPN，让两边内网主机透明互通。

- 场景 A → **传输模式**。两端是主机、做端到端保护、无需隐藏内网拓扑，传输模式开销更小，保留原 IP 头即可直达。
- 场景 B → **隧道模式**。网关代表整个内网收发，必须封装整个原始分组（含内网私有地址）并加网关公网地址的新 IP 头才能在公网路由，只有隧道模式能胜任。

判断口诀：**主机到主机用传输模式，涉及网关就用隧道模式**。

### 例题二：选 AH 还是 ESP

> 需求：通过公网传输敏感数据，要求外人**既看不到内容、又能确认未被篡改**；链路上还存在 NAT。

选 **ESP（启用其完整性/认证选项）**。理由：

- "看不到内容"=机密性，只有 ESP 能加密，AH 不行；
- "确认未被篡改"=完整性/认证，ESP 的可选 ICV 即可提供；
- 存在 NAT，AH 因认证覆盖外层 IP 头而无法穿越，ESP 则可（配合 NAT-T）。

一个 ESP 即可同时满足全部要求，无需再叠加 AH。

### 例题三：AH/ESP 安全服务辨析

判断下列说法是否正确：

1. "用 AH 保护后，攻击者无法读取分组内容。" —— **错**。AH 不加密，内容仍是明文，AH 只保证未被篡改、来源可信、非重放。
2. "ESP 一定提供机密性，但不一定提供完整性。" —— **对**。ESP 必然加密（机密性），完整性/认证是可选项（实践中通常启用）。
3. "保护 A 与 B 之间用 ESP 的双向通信，只需一个 SA。" —— **错**。SA 单向，双向需两个 SA。
4. "隧道模式下原始 IP 头会被隐藏。" —— **对**。隧道模式把整个原分组（含原 IP 头）封进载荷，外层换成新 IP 头，原头被保护/隐藏。
5. "AH 比 ESP 更适合有 NAT 的环境。" —— **错**。恰恰相反，AH 认证覆盖外层 IP 头，与 NAT 改址冲突；ESP 才适合有 NAT 的环境。

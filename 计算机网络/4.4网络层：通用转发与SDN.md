## 通用转发基本概念

### 从传统转发到通用转发

> **通用转发 (Generalized Forwarding)**
> 
> 基于分组多个头部字段的"匹配-动作"机制。传统转发只看目的IP地址查表，通用转发可对任意头部字段组合做匹配，并执行转发以外的动作（丢弃、改写、泛洪等）。

传统转发 vs 通用转发：

| 维度 | 传统转发 | 通用转发 |
|------|----------|----------|
| 匹配依据 | 目的IP地址 | 多个头部字段（链路层/网络层/传输层） |
| 动作 | 转发到输出端口 | 转发、丢弃、改写头部、泛洪、送控制器 |
| 决策逻辑 | 固定的最长前缀匹配 | 流表项可编程下发 |

```
传统转发：
  目的IP ──→ 转发表（最长前缀匹配）──→ 输出端口

通用转发：
  多字段 ──→ 流表（匹配-动作，按优先级）──→ 动作集
```

易混：传统转发的"匹配"只是目的IP的最长前缀匹配，动作只有"转发到某端口"；通用转发把匹配扩展到任意字段、把动作扩展到转发之外，二者是特例与一般化的关系。

### 通用转发能实现的网络功能

同一套"匹配-动作"机制，配不同流表项即可实现多种设备的功能：

| 功能 | 匹配字段 | 动作 |
|------|----------|------|
| 路由 | 目的IP地址 | 转发到对应输出端口 |
| 防火墙/ACL | 源/目的IP、端口、协议号 | 命中则丢弃或放行 |
| 负载均衡 | 目的服务器地址 | 改写目的地址，分发到后端 |
| NAT | 私有IP、端口 | 改写源IP和端口 |

## 匹配-动作转发机制

### 匹配-动作抽象

通用转发把每台设备抽象成一张**流表 (Flow Table)**，分组到来时按"匹配 → 动作"两步处理：

```
分组到达
   │
   ▼
┌──────────────────────────────────────────┐
│  流表（按优先级从高到低逐项匹配）           │
│  ┌────────────┬──────────┬──────────┐      │
│  │ 匹配字段    │ 动作      │ 计数器    │      │
│  ├────────────┼──────────┼──────────┤      │
│  │ 字段组合①  │ 动作①    │ 命中数①  │      │
│  │ 字段组合②  │ 动作②    │ 命中数②  │      │
│  │   ...      │   ...    │   ...    │      │
│  └────────────┴──────────┴──────────┘      │
└──────────────────────────────────────────┘
   │ 命中最高优先级表项               │ 无命中
   ▼                                  ▼
执行其动作集 + 更新计数器          默认动作（通常送控制器）
```

### 流表项结构

> **流表项 (Flow Entry)**
> 
> 由三部分组成：匹配字段、动作集、计数器。

| 组成 | 内容 | 作用 |
|------|------|------|
| 匹配字段 | 链路层/网络层/传输层头部的指定字段（可带通配符） | 决定哪些分组命中本项 |
| 动作 | 命中后执行的动作集（转发、丢弃、改写等） | 决定如何处理 |
| 计数器 | 命中分组数、命中字节数 | 统计与监控 |

注：表项还带优先级；多项命中时取优先级最高者。

### 匹配字段

OpenFlow（1.0）可匹配的头部字段跨链路层、网络层、传输层三层：

| 层 | 字段 | 典型用途 |
|----|------|----------|
| 链路层 | 入端口、源/目的 MAC、以太网类型、VLAN ID/优先级 | 二层转发、VLAN |
| 网络层 | 源/目的 IP、IP 协议号、ToS | 路由、协议过滤、QoS |
| 传输层 | 源/目的 TCP/UDP 端口 | 应用识别、防火墙 |

注：IP 地址按前缀匹配，其余字段精确匹配；任意字段可置通配符表示"不关心"。匹配只针对协议头部，不含分组负载。

匹配粒度举例：

```
精确：  目的IP = 192.168.1.100        仅完全相等者命中
前缀：  目的IP = 192.168.0.0/16       该网段全部命中
通配：  源IP=*，目的端口 = 80         所有目的端口为80的流量命中
```

### 动作

命中表项后执行其动作集，主要分三类：

| 类型 | 动作 | 说明 |
|------|------|------|
| 转发 | 转发到指定端口 | 等价传统转发 |
|  | 泛洪 / 送所有端口 | 从入端口外所有端口（泛洪）或全部端口转发 |
|  | 送控制器 | 封装上交控制器处理（如新流首包） |
| 丢弃 | 丢弃 | 无动作即丢弃，用于防火墙/限速 |
| 改写 | 改写头部字段 | 修改 MAC/IP/端口、增删 VLAN 标签、递减 TTL 等 |

注：多个动作可组合成动作集，按序执行。例如"递减 TTL → 改写目的 MAC 为下一跳 → 转发到输出端口"，即模拟一次传统路由转发。

## OpenFlow协议

### OpenFlow 在 SDN 中的位置

> **OpenFlow**
> 
> 控制器与交换机之间的标准协议，运行在控制平面与数据平面之间，定义了流表的下发与状态上报格式。

```
        ┌──────────────────┐
        │   SDN 控制器      │   集中式控制平面
        └────────┬─────────┘
                 │ OpenFlow（南向接口）
─────────────────┼───────────────── 控制平面 / 数据平面分界
                 │
        ┌────────┴─────────┐
        │  OpenFlow 交换机  │   可编程数据平面
        │  （流表）         │   只按下发的流表项做匹配-动作
        └──────────────────┘
```

注：控制器掌握全网视图并计算转发逻辑，交换机本身不运行路由算法，只执行下发的流表项。

### OpenFlow 消息类型

按方向分三类：

控制器 → 交换机：

| 消息 | 功能 |
|------|------|
| FLOW_MOD | 增删改流表项（控制器下发转发逻辑的主要手段） |
| PACKET_OUT | 指示交换机从指定端口发出某分组 |
| BARRIER | 确保此前消息执行完毕，用于同步 |

交换机 → 控制器：

| 消息 | 功能 |
|------|------|
| PACKET_IN | 上交未命中任何表项的分组（新流首包） |
| FLOW_REMOVED | 表项因超时或被删而移除时通知 |
| PORT_STATUS | 端口状态变化通知 |

对称消息：

| 消息 | 功能 |
|------|------|
| HELLO | 建立连接、协商版本 |
| ECHO | 心跳保活 |
| ERROR | 报告错误 |

### 流表项匹配逻辑

交换机按优先级从高到低逐项匹配，命中即停：

```python
def flow_table_lookup(packet):
    for entry in sorted(flow_table, key=lambda e: e.priority, reverse=True):
        if matches(packet, entry.match_fields):
            execute_actions(entry.actions)
            update_counters(entry)
            return
    send_to_controller(packet)   # 无命中时的默认动作
```

用优先级 + 前缀，可在通用机制上复现 IP 路由的最长前缀匹配：

```
优先级 100:  192.168.1.0/24 → 端口1
优先级  50:  192.168.0.0/16 → 端口2
优先级  10:  0.0.0.0/0      → 控制器

目的 192.168.1.100 → 命中优先级100项 → 端口1
```

注：更具体的前缀给更高优先级，命中即停，效果等同最长前缀匹配。

## 软件定义网络架构

### SDN核心理念

> **软件定义网络 (Software-Defined Networking, SDN)**
> 
> 通过将网络控制平面从数据平面中分离，实现网络的集中控制和灵活编程。

**SDN 三大特征**：

控制平面与数据平面分离 —— 传统路由器自带控制平面（分布式跑路由算法），SDN 把控制逻辑上提到独立控制器，交换机只剩数据平面：

```
传统路由器                SDN
┌──────────┐            ┌──────────┐
│ 控制平面  │ 分布式      │ SDN 控制器│ 集中式
│ 数据平面  │            └────┬─────┘
└──────────┘                 │ 南向接口
                        ┌────┴─────┐
                        │ 数据平面  │ 只做转发
                        └──────────┘
```

集中式控制 —— 控制器掌握全局拓扑，在全网范围统一计算路径，避免分布式算法的收敛与一致性问题。

开放可编程接口：

- **南向接口**：控制器对交换机，如 OpenFlow
- **北向接口**：网络应用对控制器，常用 RESTful API
- **东西向接口**：多控制器间通信，用于集群

### SDN 控制器分层架构

```
┌──────────────────────────────────┐
│ 网络应用层  路由 | 防火墙 | 负载均衡 │  业务逻辑
├──────────────────────────────────┤
│ 北向接口    RESTful API            │  应用编程接口
├──────────────────────────────────┤
│ 控制器核心  拓扑管理|流表管理|状态维护│  集中控制逻辑
├──────────────────────────────────┤
│ 南向接口    OpenFlow | NETCONF      │  设备通信接口
├──────────────────────────────────┤
│ 网络设备层  交换机 | 虚拟交换机      │  数据平面
└──────────────────────────────────┘
```

主流 SDN 控制器：

| 控制器 | 语言 | 特点 | 典型场景 |
|--------|------|------|----------|
| OpenDaylight | Java | 模块化、插件丰富 | 企业级部署 |
| ONOS | Java | 高可用、支持集群 | 运营商网络 |
| Floodlight | Java | 轻量、易扩展 | 小型网络、教学 |
| Ryu | Python | 上手简单、API 丰富 | 原型与研发 |

### SDN 应用举例

控制器之上可编写网络应用，借南向接口下发流表。两个典型例子：

- **拓扑发现**：周期性发 LLDP 探测包，收集邻居信息构建全网拓扑图，供其他应用查询。
- **最短路径路由**：收到 PACKET_IN（新流首包）后，解析源/目的地址，用 Dijkstra 算出路径，沿途交换机各下发一条流表项。

## SDN应用与发展

### 典型部署场景

| 场景 | 痛点 | SDN 的作用 |
|------|------|-----------|
| 数据中心网络 | 配置复杂、扩展难 | 租户隔离、动态负载均衡、故障快速重路由 |
| 广域网优化 | 链路利用率不均 | 基于全局流量统一优化路径、多路径均衡 |
| 网络功能虚拟化（NFV） | 功能绑定专用硬件 | SDN 负责连接、NFV 提供虚拟防火墙/负载均衡等功能 |

注：SDN 与 NFV 常配合使用——SDN 管"流量怎么走"，NFV 管"用什么软件功能处理流量"。

### SDN 的主要挑战

| 问题 | 表现 | 缓解手段 |
|------|------|---------|
| 性能 | 控制器是集中瓶颈；新流首包需上交控制器，增加首包时延 | 控制器集群、主动（proactive）下发流表 |
| 可靠性 | 控制器单点故障影响全网 | 主备控制器、多控制器分布式共识 |
| 安全 | 控制器成为集中攻击目标 | 通信认证、应用间权限隔离 |

注：性能上常区分两种流表下发方式——**reactive**（首包触发 PACKET_IN，控制器再下发）时延高但表项少；**proactive**（预先下发）首包无需上交，时延低。

**标准化**：

| 组织 | 标准 | 领域 |
|------|------|------|
| ONF | OpenFlow | 南向接口 |
| IETF | I2RS | 路由系统接口 |
| ITU-T | SDN 架构建议 | 体系结构 |

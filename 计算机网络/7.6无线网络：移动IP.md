## 移动IP基本概念

### 移动IP的需求

> **移动性问题**：传统IP协议基于位置的地址分配机制，当移动节点在不同网络间移动时，需要更改IP地址，导致通信中断。

**传统IP的局限**：
- IP地址既标识设备身份，又标识网络位置
- 更改IP地址会导致上层连接中断
- 需要一种机制支持移动节点保持IP地址不变

**移动IP的目标**：
- **透明性**：移动对上层应用透明
- **兼容性**：与现有IP协议兼容
- **效率性**：最小化路由开销
- **安全性**：防止地址欺骗

### 移动IP核心概念

> **定义**：移动IP是一种允许移动节点在改变接入点的情况下保持永久IP地址，并维持正在进行的通信的协议。

**关键实体**：

1. **移动节点（Mobile Node, MN）**
   - 可以改变接入点的主机或路由器
   - 拥有永久归属地址
   - 移动时获得转交地址

2. **归属代理（Home Agent, HA）**
   - 位于移动节点归属网络的路由器
   - 维护移动节点的当前位置信息
   - 截获发往移动节点的数据包并转发

3. **外部代理（Foreign Agent, FA）**
   - 位于外部网络的路由器
   - 为访问的移动节点提供路由服务
   - 移动IPv6中可以不需要FA

4. **通信节点（Correspondent Node, CN）**
   - 与移动节点通信的任意节点
   - 可以是固定节点或移动节点

**关键地址**：

| 地址类型 | 说明 | 特点 |
|---------|------|------|
| 归属地址（Home Address） | 移动节点的永久地址 | 标识移动节点身份，属于归属网络 |
| 转交地址（Care-of Address, CoA） | 移动节点在外部网络的临时地址 | 标识当前位置，可以是FA地址或共址转交地址 |
| 归属网络（Home Network） | 移动节点的永久网络 | 由归属地址的网络前缀确定 |
| 外部网络（Foreign Network） | 移动节点访问的网络 | 非归属网络的任何网络 |

## 移动IPv4工作原理

### 移动IPv4基本流程

**1. 代理发现（Agent Discovery）**

移动节点发现可用的归属代理或外部代理：

```
方法1：代理广告（Agent Advertisement）
- FA/HA周期性广播ICMP Router Advertisement消息
- 包含移动性扩展信息
- MN被动监听获取代理信息

方法2：代理请求（Agent Solicitation）
- MN主动发送ICMP Router Solicitation消息
- 代理响应广告消息
- 加快代理发现过程
```

**代理广告消息格式**：
- 基于ICMP Router Advertisement
- 添加移动性扩展字段
- 包含转交地址、注册生命期等信息

**2. 注册（Registration）**

移动节点向归属代理注册当前位置：

```
注册流程：
MN → FA: 注册请求（Registration Request）
FA → HA: 转发注册请求
HA → FA: 注册应答（Registration Reply）
FA → MN: 转发注册应答
```

**注册请求消息包含**：
- 归属地址
- 转交地址
- 注册生命期
- 身份认证信息

**注册应答消息包含**：
- 接受/拒绝状态码
- 剩余生命期
- 身份认证信息

**3. 数据传输（Tunneling）**

使用隧道技术转发数据包：

**从CN到MN的数据传输**：
```
1. CN发送数据包到MN的归属地址
   源地址: CN
   目的地址: MN归属地址

2. HA截获数据包（通过代理ARP）
   
3. HA封装数据包（IP-in-IP隧道）
   外层IP头:
     源地址: HA
     目的地址: CoA
   内层IP头:
     源地址: CN
     目的地址: MN归属地址

4. FA接收并解封装
   
5. FA转发给MN
   源地址: CN
   目的地址: MN归属地址
```

**从MN到CN的数据传输**：
```
标准路由（Standard Routing）：
MN → FA → 常规路由 → CN
源地址: MN归属地址
目的地址: CN

反向隧道（Reverse Tunneling）：
MN → FA → HA → CN
解决入口过滤问题
```

### 移动IPv4隧道技术

**1. IP-in-IP隧道**

> **定义**：将原始IP数据包封装在新的IP数据包中，形成隧道传输。

**封装格式**：
```
┌─────────────────────────────────────────────────────────┐
  IP-in-IP隧道封装
├─────────────────────────────────────────────────────────┤
  外层IP头（20字节）
  ├─ 源地址: HA地址
  ├─ 目的地址: CoA
  └─ 协议: 4（IP-in-IP）
├─────────────────────────────────────────────────────────┤
  内层IP头（20字节）
  ├─ 源地址: CN地址
  ├─ 目的地址: MN归属地址
  └─ 协议: 6（TCP）或17（UDP）
├─────────────────────────────────────────────────────────┤
  传输层头部（TCP/UDP）
├─────────────────────────────────────────────────────────┤
  应用层数据
└─────────────────────────────────────────────────────────┘
```

**2. 最小封装（Minimal Encapsulation）**

减少隧道开销：
- 移除外层IP头的某些字段
- 减少20字节到8-12字节
- 提高传输效率

**3. GRE隧道（Generic Routing Encapsulation）**

通用路由封装：
- 更灵活的封装协议
- 支持多种网络层协议
- 添加GRE头部（4字节或更多）

### 三角路由问题

> **三角路由问题**：从CN到MN的数据包必须经过HA，即使CN和MN在同一网络，也要绕道HA，导致路由次优。

**问题示意**：
```
CN -----------------> HA -----------------> FA --> MN
    直接距离短              绕道距离长
    
实际路径: CN → HA → FA → MN
理想路径: CN → FA → MN
```

**性能影响**：
- 增加端到端延迟
- 增加网络负载
- HA成为瓶颈
- 浪费网络带宽

**路由优化（Route Optimization）**：

1. **绑定更新（Binding Update）**
   - MN直接通知CN当前的CoA
   - CN维护绑定缓存（Binding Cache）
   - 后续数据包直接发往CoA

2. **绑定缓存（Binding Cache）**
   - CN维护MN的归属地址到CoA的映射
   - 包含绑定生命期
   - 定期更新和超时管理

3. **优化后的数据传输**：
```
CN直接发送到MN:
源地址: CN
目的地址: CoA
路由头: 包含MN归属地址

MN回复CN:
源地址: MN归属地址  
目的地址: CN
正常路由
```

## 移动IPv6

### 移动IPv6特点

> **移动IPv6优势**：相比移动IPv4，移动IPv6在协议设计中原生集成了移动性支持，更加高效和简洁。

**主要改进**：

| 特性 | 移动IPv4 | 移动IPv6 |
|-----|---------|----------|
| 外部代理 | 必需 | 不需要（共址转交地址） |
| 路由优化 | 可选扩展 | 内置支持 |
| 地址自动配置 | 需要FA支持 | 无状态地址自动配置 |
| 安全性 | 可选 | 强制IPsec |
| 三角路由 | 默认存在 | 默认优化 |
| 隧道开销 | IP-in-IP(20字节) | 路由头(24字节) |

### 移动IPv6工作机制

**1. 转交地址获取**

移动节点自动配置转交地址：
```
1. MN接收路由器通告（Router Advertisement）
   获取网络前缀信息
   
2. 无状态地址自动配置（SLAAC）
   CoA = 网络前缀 + 接口标识符
   接口标识符从MAC地址生成（EUI-64）
   
3. 重复地址检测（DAD）
   确保CoA唯一性
```

**2. 绑定更新过程**

**向归属代理注册**：
```
MN → HA: Binding Update (BU)
  - 归属地址
  - 转交地址
  - 生命期
  - 序列号
  - Home Registration (H) 标志

HA → MN: Binding Acknowledgement (BA)
  - 状态码
  - 序列号
  - 剩余生命期
```

**向通信节点注册**：
```
MN → CN: Binding Update
  - 归属地址
  - 转交地址
  - 生命期
  - 序列号

CN → MN: Binding Acknowledgement
  - 状态码
  - 绑定刷新建议时间
```

**3. 数据包交换**

**首次通信（通过HA）**：
```
CN → HA → MN:
  IPv6头:
    源地址: CN
    目的地址: MN归属地址
  路由头类型2:
    下一跳: CoA

MN → CN:
  IPv6头:
    源地址: MN归属地址
    目的地址: CN
  目的选项头:
    归属地址选项
```

**路由优化后**：
```
CN → MN直接通信:
  IPv6头:
    源地址: CN
    目的地址: CoA
  路由头类型2:
    最终目的地址: MN归属地址

MN → CN:
  IPv6头:
    源地址: CoA
    目的地址: CN
  目的选项头:
    归属地址: MN归属地址
```

### 移动IPv6扩展头

**路由头类型2（Routing Header Type 2）**：
```
┌─────────────────────────────────────────────────────────┐
  路由头类型2 - 用于移动IPv6
├─────────────────────────────────────────────────────────┤
  下一个头部 (8位) - 后续扩展头或上层协议
├─────────────────────────────────────────────────────────┤
  头部扩展长度 (8位) - 值为2（16字节）
├─────────────────────────────────────────────────────────┤
  路由类型 (8位) - 值为2
├─────────────────────────────────────────────────────────┤
  剩余段数 (8位) - 值为1
├─────────────────────────────────────────────────────────┤
  保留 (32位) - 必须为0
├─────────────────────────────────────────────────────────┤
  归属地址 (128位) - MN的归属地址
└─────────────────────────────────────────────────────────┘
```

**目的选项头（Destination Options Header）**：
```
包含归属地址选项：
- 选项类型: 201（0xC9）
- 选项长度: 16字节
- 归属地址: 128位IPv6地址
```

## 移动IP性能分析

### 隧道开销分析

**封装开销计算**：

> **封装开销**：隧道技术增加的额外头部字节数。

| 隧道类型 | 额外开销 | 总开销（含内层IP头） |
|---------|---------|-------------------|
| IPv4-in-IPv4 | 20字节 | 40字节 |
| 最小封装 | 8-12字节 | 28-32字节 |
| GRE隧道 | 24字节 | 44字节 |
| IPv6路由头 | 24字节（路由头类型2） | 64字节（含IPv6头40字节） |

**带宽效率计算**：

$$\text{带宽效率} = \frac{\text{有效载荷}}{\text{有效载荷} + \text{隧道开销}} \times 100\%$$

**示例**：
- 有效载荷：1460字节（TCP MSS）
- IPv4-in-IPv4隧道：$\frac{1460}{1460 + 20} = 98.65\%$
- IPv6路由头：$\frac{1460}{1460 + 24} = 98.38\%$

### 延迟分析

**三角路由延迟**：

$$T_{\text{三角路由}} = T_{CN \to HA} + T_{HA \to FA} + T_{FA \to MN}$$

**路由优化延迟**：

$$T_{\text{优化}} = T_{CN \to FA} + T_{FA \to MN}$$

**延迟改善**：

$$\Delta T = T_{\text{三角路由}} - T_{\text{优化}}$$

### 注册开销

**注册消息大小**：

| 消息类型 | IPv4大小 | IPv6大小 |
|---------|---------|----------|
| 注册请求/绑定更新 | 约70字节 | 约100字节 |
| 注册应答/绑定确认 | 约70字节 | 约80字节 |
| 总往返开销 | 约140字节 | 约180字节 |

**注册频率**：

$$\text{注册次数/小时} = \frac{3600}{\text{注册生命期（秒）}}$$

**注册带宽开销**：

$$\text{带宽开销} = \frac{\text{注册消息大小} \times \text{注册次数}}{\text{时间}}$$

## 移动IP计算例题

### 例题1：隧道开销计算

> **例题**：移动节点使用移动IPv4通信，原始IP数据包大小为1500字节（包含20字节IP头、20字节TCP头、1460字节数据）。计算使用IP-in-IP隧道后的总开销和带宽利用率。

**解题步骤**：

1. **原始数据包组成**：
   - IP头：20字节
   - TCP头：20字节
   - 应用数据：1460字节
   - 总计：1500字节

2. **隧道封装后**：
   - 外层IP头：20字节
   - 内层IP头：20字节
   - TCP头：20字节
   - 应用数据：1460字节
   - **总计：1520字节**

3. **开销分析**：
   - 额外开销：20字节（外层IP头）
   - 相对增加：$\frac{20}{1500} \times 100\% = 1.33\%$

4. **带宽利用率**：
   $$\text{带宽利用率} = \frac{1460}{1520} \times 100\% = 96.05\%$$

**答案**：隧道增加20字节开销，数据包增大到1520字节，带宽利用率为96.05%。

### 例题2：三角路由延迟计算

> **例题**：移动节点从归属网络移动到外部网络，与通信节点CN通信。已知：
> - CN到HA的RTT：40ms
> - HA到FA的RTT：30ms
> - CN到FA的RTT：15ms
> - FA到MN的延迟：2ms
> 
> 计算三角路由和路由优化的单向延迟，并分析性能改善。

**解题步骤**：

1. **三角路由单向延迟**：
   $$T_{\text{三角}} = \frac{\text{RTT}_{CN \to HA}}{2} + \frac{\text{RTT}_{HA \to FA}}{2} + T_{FA \to MN}$$
   $$T_{\text{三角}} = \frac{40}{2} + \frac{30}{2} + 2 = 20 + 15 + 2 = 37\text{ms}$$

2. **路由优化单向延迟**：
   $$T_{\text{优化}} = \frac{\text{RTT}_{CN \to FA}}{2} + T_{FA \to MN}$$
   $$T_{\text{优化}} = \frac{15}{2} + 2 = 7.5 + 2 = 9.5\text{ms}$$

3. **性能改善**：
   $$\Delta T = 37 - 9.5 = 27.5\text{ms}$$
   $$\text{改善百分比} = \frac{27.5}{37} \times 100\% = 74.3\%$$

**答案**：
- 三角路由延迟：37ms
- 路由优化延迟：9.5ms
- 延迟减少：27.5ms（74.3%改善）

### 例题3：注册开销分析

> **例题**：移动节点使用移动IPv6，注册生命期为60秒，每次绑定更新消息100字节（含IPv6头），绑定确认80字节。移动节点同时向1个HA和3个CN注册。计算每小时的注册带宽开销。

**解题步骤**：

1. **注册频率**：
   $$\text{每小时注册次数} = \frac{3600}{60} = 60\text{次}$$

2. **单次注册开销**：
   - 向HA：BU(100) + BA(80) = 180字节
   - 向3个CN：3 × [BU(100) + BA(80)] = 3 × 180 = 540字节
   - **单次总开销**：180 + 540 = 720字节

3. **每小时总开销**：
   $$\text{总开销} = 720 \times 60 = 43,200\text{字节} = 43.2\text{KB}$$

4. **平均带宽消耗**：
   $$\text{平均带宽} = \frac{43,200 \times 8}{3600} = 96\text{bps}$$

**答案**：每小时注册开销为43.2KB，平均带宽消耗为96bps。

### 例题4：移动场景综合分析

> **例题**：某移动用户在高铁上移动，速度120km/h，使用4G网络。基站覆盖半径5km，移动IPv4隧道开销20字节。用户观看视频流，码率2Mbps。计算：
> 1. 平均切换频率
> 2. 因隧道增加的额外带宽消耗
> 3. 如果视频服务器与用户同城，三角路由增加的额外延迟（HA在另一城市，额外RTT 40ms）

**解题步骤**：

**1. 平均切换频率**：
- 基站直径：$2 \times 5 = 10\text{km}$
- 移动速度：120 km/h
- 切换频率：
  $$f = \frac{120}{10} = 12\text{次/小时} = 0.2\text{次/分钟}$$
  平均每5分钟切换一次

**2. 隧道额外带宽**：
- 视频码率：2 Mbps = 2,000,000 bps = 250,000 字节/秒
- 假设MTU 1500字节，有效载荷约1460字节
- 每秒数据包数：$\frac{250,000}{1460} \approx 171$包/秒
- 每包隧道开销：20字节
- 额外带宽：$171 \times 20 \times 8 = 27,360\text{ bps} \approx 27\text{ Kbps}$
- 额外带宽比例：$\frac{27}{2000} = 1.35\%$

**3. 三角路由额外延迟**：
- 三角路由路径：用户 → HA（跨城） → 视频服务器（本城）
- 本城直连延迟：约5ms（估算）
- HA跨城RTT：40ms，单向20ms
- 三角路由总延迟：$20 + 5 = 25\text{ms}$
- 额外延迟：$25 - 5 = 20\text{ms}$

**答案**：
1. 平均每5分钟切换一次（12次/小时）
2. 隧道增加约27 Kbps额外带宽（1.35%开销）
3. 三角路由增加约20ms延迟

### 例题5：绑定缓存容量规划

> **例题**：一个内容分发网络（CDN）边缘服务器需要支持移动IPv6客户端。预计同时服务10,000个移动用户，每个绑定缓存条目128字节。绑定生命期300秒，每次更新需要180字节（BU+BA）。计算：
> 1. 绑定缓存所需内存
> 2. 每秒处理的绑定更新数量
> 3. 绑定更新的带宽消耗

**解题步骤**：

**1. 绑定缓存内存需求**：
$$\text{内存} = 10,000 \times 128 = 1,280,000\text{字节} = 1.22\text{ MB}$$

**2. 每秒绑定更新数量**：
- 生命期300秒，则每个用户每300秒更新一次
- 平均每秒更新数：
  $$\text{更新率} = \frac{10,000}{300} \approx 33.33\text{次/秒}$$

**3. 绑定更新带宽**：
- 每次更新180字节
- 带宽消耗：
  $$BW = 33.33 \times 180 \times 8 = 47,995\text{ bps} \approx 48\text{ Kbps}$$

**答案**：
1. 需要约1.22 MB内存存储绑定缓存
2. 平均每秒处理33次绑定更新
3. 绑定更新消耗约48 Kbps带宽

## 移动IP安全性

### 安全威胁

**1. 注册劫持**
- 攻击者伪造绑定更新
- 窃取移动节点流量
- 重定向到恶意节点

**2. 中间人攻击**
- 拦截注册消息
- 篡改转交地址
- 窃听通信内容

**3. 重放攻击**
- 重放旧的注册消息
- 导致路由错误
- 拒绝服务攻击

### 安全机制

**移动IPv4安全**：

1. **注册认证**
   - MN-HA认证（必需）
   - MN-FA认证（可选）
   - FA-HA认证（可选）

2. **认证扩展**
   - 基于共享密钥
   - MD5消息摘要
   - 序列号防重放

**认证扩展格式**：
```
- 类型：32或33
- 长度：可变
- SPI（安全参数索引）：4字节
- 认证数据：16字节（MD5）
```

**移动IPv6安全**：

1. **强制IPsec**
   - MN-HA之间必须使用IPsec ESP
   - 保护绑定更新消息
   - 提供机密性和完整性

2. **返回可达性测试（Return Routability Test）**
   - 验证CN能够到达CoA
   - 防止地址欺骗
   - Home Test + Care-of Test

**返回可达性测试流程**：
```
1. MN → CN: HoTI (Home Test Init)（经HA）
2. MN → CN: CoTI (Care-of Test Init)（直接）
3. CN → MN: HoT (Home Test)（经HA）
4. CN → MN: CoT (Care-of Test)（直接）
5. MN生成绑定管理密钥（Kbm）
   Kbm = Hash(Home Keygen Token | Care-of Keygen Token)
6. MN → CN: BU（使用Kbm认证）
```

## 移动IP实际应用

### 典型应用场景

**1. 移动办公**
- 员工在不同办公地点移动
- 保持VPN连接不中断
- 透明的网络切换

**2. 车载网络**
- 车辆高速移动
- 频繁切换基站
- 实时导航和娱乐服务

**3. 移动视频监控**
- 移动摄像头实时传输
- 保持监控中心连接
- 支持双向通信

**4. 物联网设备**
- 可移动传感器
- 资产追踪
- 移动医疗设备

### 部署挑战

**1. 技术挑战**
- 切换延迟影响实时应用
- 三角路由效率问题
- 防火墙和NAT穿越困难

**2. 运营挑战**
- 需要网络基础设施支持
- 归属代理部署和管理
- 跨运营商移动性

**3. 性能优化**
- 快速切换技术（FMIPv6）
- 分层移动管理（HMIPv6）
- 代理移动IP（PMIPv6）

### 改进协议

**1. 快速移动IPv6（FMIPv6）**
- 预测性切换
- 减少切换延迟
- 提前建立隧道

**2. 分层移动IPv6（HMIPv6）**
- 引入移动锚点（MAP）
- 本地移动不通知HA
- 减少信令开销

**3. 代理移动IPv6（PMIPv6）**
- 网络侧实现移动性
- 移动节点无需协议栈修改
- 简化部署

## 性能对比总结

| 指标 | 移动IPv4 | 移动IPv6 | PMIPv6 |
|-----|---------|----------|--------|
| 外部代理 | 需要 | 不需要 | 需要（MAG） |
| 路由优化 | 可选 | 内置 | 不支持 |
| 切换延迟 | 较高（>1s） | 中等（500-800ms） | 较低（<300ms） |
| 信令开销 | 中等 | 较高 | 较低 |
| 终端修改 | 需要 | 需要 | 不需要 |
| 安全性 | 基本认证 | IPsec | IPsec |
| 部署复杂度 | 中等 | 中等 | 较低 |

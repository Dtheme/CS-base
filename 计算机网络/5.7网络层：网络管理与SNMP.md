## 网络管理基础

### 什么是网络管理

> **网络管理 (Network Management)**
>
> 部署、集成和协调硬件、软件与人力资源，以监测、测试、轮询、配置、分析、评价和控制网络资源，使其在合理成本下满足实时性、运行性能与服务质量要求。

一个网络由成千上万的设备（路由器、交换机、主机、链路）组成，分布在不同地点、来自不同厂商。网络管理的目标是用一套统一的机制远程监测设备状态、收集运行数据、下发配置、并在异常时收到通知。

### 网络管理的功能：FCAPS

ISO 把网络管理划分为五大功能领域，缩写 **FCAPS**：

| 功能 | 含义 | 典型任务 |
|------|------|---------|
| **F** 故障管理 (Fault) | 检测、隔离、修复故障 | 链路中断告警、故障定位 |
| **C** 配置管理 (Configuration) | 跟踪与下发设备配置 | 拓扑发现、配置备份与变更 |
| **A** 计费管理 (Accounting) | 统计资源使用量 | 流量统计、配额、账单 |
| **P** 性能管理 (Performance) | 量化与优化网络性能 | 利用率监控、阈值告警、容量规划 |
| **S** 安全管理 (Security) | 控制访问、保护资源 | 访问控制、审计、密钥管理 |

注：FCAPS 只是功能分类，并不规定具体协议。Internet 体系中承担这些功能的标准协议是 **SNMP**。

## 网络管理框架

### 三个核心组件

Internet 网络管理框架由三部分构成：

```
┌──────────────────────────────────────────────┐
│              管理服务器                         │
│         (Managing Server / NMS)                │
│   • 运行网络管理应用 (监控、告警、报表)          │
│   • 由网络管理员操作                            │
└───────────────────┬──────────────────────────┘
                    │  SNMP (运行在 UDP 之上)
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ 被管设备 │ │ 被管设备 │ │ 被管设备 │
   │┌───────┐│ │┌───────┐│ │┌───────┐│
   ││ 代理  ││ ││ 代理  ││ ││ 代理  ││ ← SNMP Agent
   │├───────┤│ │├───────┤│ │├───────┤│
   ││  MIB  ││ ││  MIB  ││ ││  MIB  ││ ← 被管对象的数据
   │└───────┘│ │└───────┘│ │└───────┘│
   └─────────┘ └─────────┘ └─────────┘
```

- **管理服务器 (managing server)**：网管中心的应用进程，通常有管理员监管。它发起对设备数据的查询和配置。
- **被管设备 (managed device)**：网络中受管理的设备（路由器、交换机、主机等），其内部包含若干**被管对象 (managed object)**，如接口状态、转发表项、协议计数器。
- **管理代理 (management agent)**：运行在被管设备上的进程，负责把本地被管对象的信息汇报给管理服务器，并执行服务器下发的动作。
- **MIB (Management Information Base)**：被管对象信息的集合，是代理与管理服务器之间交换数据的"数据模型"。

### 工作模式

管理服务器与代理通过 SNMP 交互，有两种基本模式：

```
请求-响应 (轮询):                     主动通告 (Trap):

管理服务器          代理              管理服务器          代理
    │  GetRequest    │                   │                 │
    │───────────────>│                   │   事件发生      │
    │                │ 读取MIB           │<────────────────│
    │   Response     │                   │   Trap          │
    │<───────────────│                   │ (无需请求)      │
```

- **轮询 (polling)**：管理服务器周期性地向代理发请求，读取或设置 MIB 中的对象值。
- **陷阱 (trap)**：代理在检测到重要事件（如接口故障）时，无需等待请求即主动向管理服务器发送通知。

## SNMP协议

### 概述

> **SNMP (Simple Network Management Protocol)**
>
> 简单网络管理协议，是 Internet 标准网络管理框架中管理服务器与代理之间的应用层协议。SNMP 报文承载于 **UDP** 之上。

端口约定：

- 代理监听 **UDP 161**：接收管理服务器的请求 (Get/GetNext/GetBulk/Set)。
- 管理服务器监听 **UDP 162**：接收代理发来的 Trap/Inform 通告。

注：因运行于 UDP，SNMP 本身不保证可靠传输；请求丢失需由管理应用自行重传。

### SNMP PDU 类型

SNMP 定义了 7 种协议数据单元 (PDU)，体现两类交互（请求-响应、主动通告）：

| PDU | 发送方 | 作用 |
|-----|--------|------|
| **GetRequest** | 管理服务器 → 代理 | 获取一个或多个被管对象的值 |
| **GetNextRequest** | 管理服务器 → 代理 | 获取 MIB 树中（字典序）下一个对象的值 |
| **GetBulkRequest** | 管理服务器 → 代理 | 一次批量获取大块数据（v2 引入） |
| **SetRequest** | 管理服务器 → 代理 | 设置一个或多个被管对象的值 |
| **Response** | 代理 → 管理服务器 | 对上述请求的应答，携带值或错误状态 |
| **Trap** | 代理 → 管理服务器 | 主动上报事件，无需确认 |
| **InformRequest** | 管理服务器 → 管理服务器 | 服务器间通告（v2 引入），需确认 |

易混：

- **GetRequest vs GetNextRequest**：Get 按你给的 OID 取值；GetNext 取该 OID 在 MIB 树中的**下一个**对象。GetNext 不需要预先知道完整 OID，常用来逐项遍历一张表（即 `snmpwalk` 的原理）。
- **Trap vs 请求-响应**：Trap 由代理**主动**发起、不需确认（v2 的 InformRequest 需确认）；其余 PDU 都是管理服务器发起的轮询。

常见 Trap 事件：`coldStart`（冷启动）、`warmStart`（热重启）、`linkDown`/`linkUp`（接口状态变化）、`authenticationFailure`（认证失败）。

### SNMP 报文格式

SNMPv1/v2c 报文由版本号、团体名 (community) 和 PDU 三部分组成：

```
┌──────────────┬───────────────┬───────────────────────────────┐
│   Version    │   Community   │            PDU                 │
│  (版本号)     │   (团体名)     │                                │
└──────────────┴───────────────┴───────────────────────────────┘
                                 │
        ┌────────────────────────┴────────────────────────┐
        ▼                                                  ▼
┌──────────┬──────────────┬─────────────┬──────────────────────┐
│ PDU类型  │  Request ID  │ Error Status │  Variable Bindings   │
│          │ (请求标识)    │ Error Index  │  (变量绑定表)         │
└──────────┴──────────────┴─────────────┴──────────────────────┘
                                          │
                                  ┌───────┴────────┐
                                  │  OID  │  Value  │ ← 一组对象名/值对
                                  └───────┴────────┘
```

- **Request ID**：标识请求，使响应能与请求配对。
- **Error Status / Error Index**：仅在 Response 中有意义，指示出错类型及出错的变量位置。
- **Variable Bindings**：一组 (OID, Value) 对，是 SNMP 传递数据的核心载体。读请求中 Value 部分留空，由代理在响应中填入。

报文采用 ASN.1 的 **BER (Basic Encoding Rules)** 编码。

注：GetBulk 的 PDU 用 `Non-repeaters` 和 `Max-repetitions` 两个字段替换了 Error Status/Error Index，分别指定"只取一次的对象数"和"重复取值的最大次数"。

## 管理信息库MIB与SMI

### MIB 与 SMI 的关系

- **MIB (管理信息库)**：被管对象的集合，可看作一个虚拟的信息数据库。每个被管对象有唯一名字 (OID) 和取值。
- **SMI (Structure of Management Information)**：定义 MIB 的**数据描述语言**，规定如何命名对象、用哪些数据类型描述对象。SMI 基于 ASN.1。

注：易混。SMI 是"语法/规则"，MIB 是"按该规则写出的数据定义"。二者关系类似"语言规范"与"用该语言写的程序"。

SMI 定义的基本数据类型包括：`INTEGER`、`OCTET STRING`、`OBJECT IDENTIFIER`、`Counter32`/`Counter64`（只增计数器）、`Gauge32`（可增可减的量度）、`TimeTicks`（时间，以 1/100 秒为单位）、`IpAddress` 等。

### 对象标识符 OID 与命名树

每个被管对象由一个**对象标识符 (OID)** 唯一命名。OID 是全局 ISO/ITU 命名树中的一条路径，从根逐级用数字标识：

```
                        root
            ┌─────────────┼─────────────┐
         ccitt(0)       iso(1)      joint(2)
                          │
                       org(3)
                          │
                       dod(6)
                          │
                   internet(1)          ← 1.3.6.1
            ┌────────────┼────────────┐
       directory(1)   mgmt(2)      private(4)
                          │              │
                     mib-2(1)       enterprises(1)
                          │              │
        ┌────────┬────────┼─────┐   厂商私有 MIB
    system(1) interfaces(2) ip(4) tcp(6) ... (各厂商分支)
        │
    sysDescr(1)  ← OID: 1.3.6.1.2.1.1.1
```

OID 的两种写法（以系统描述为例）：

- 数字形式：`1.3.6.1.2.1.1.1.0`
- 符号形式：`iso.org.dod.internet.mgmt.mib-2.system.sysDescr.0`

注：末尾的 `.0` 表示标量对象的唯一实例；表中对象则用索引（如接口序号）作为后缀。

### MIB-II 标准对象组

标准 MIB（MIB-II，RFC 1213）在 `1.3.6.1.2.1` 下定义了常用对象组：

| 组 | OID | 内容 |
|----|-----|------|
| **system** | 1.3.6.1.2.1.1 | 设备描述、运行时间、联系人、位置等 |
| **interfaces** | 1.3.6.1.2.1.2 | 接口数量及接口表（速率、状态、收发字节/包数等） |
| **ip** | 1.3.6.1.2.1.4 | IP 层统计、路由表、ARP 表 |
| **icmp** | 1.3.6.1.2.1.5 | ICMP 消息统计 |
| **tcp** | 1.3.6.1.2.1.6 | TCP 连接与统计 |
| **udp** | 1.3.6.1.2.1.7 | UDP 统计 |
| **snmp** | 1.3.6.1.2.1.11 | SNMP 自身的运行统计 |

system 组的常用标量对象：

| 对象 | OID 后缀 | 含义 | 访问 |
|------|---------|------|------|
| sysDescr | .1 | 系统描述（型号、OS 版本） | 只读 |
| sysObjectID | .2 | 厂商标识 OID | 只读 |
| sysUpTime | .3 | 系统启动以来的时间 | 只读 |
| sysContact | .4 | 联系人 | 读写 |
| sysName | .5 | 系统名 | 读写 |
| sysLocation | .6 | 设备位置 | 读写 |

接口表 (ifTable) 是典型的 MIB 表对象，每个接口一行，列包括 `ifIndex`（索引）、`ifDescr`、`ifType`、`ifSpeed`、`ifAdminStatus`/`ifOperStatus`（管理/运行状态）、`ifInOctets`/`ifOutOctets`（收发字节数）等。性能监控正是周期性读取这些计数器并做差分。

### 用 SMI 定义对象（示例）

一个被管对象在 MIB 中用 `OBJECT-TYPE` 宏定义：

```
sysUpTime OBJECT-TYPE
    SYNTAX      TimeTicks          -- 数据类型
    MAX-ACCESS  read-only          -- 访问权限
    STATUS      current
    DESCRIPTION "系统启动以来经过的时间，以 1/100 秒为单位"
    ::= { system 3 }               -- 在树中的位置: system.3
```

`SYNTAX` 指定数据类型，`MAX-ACCESS` 指定读写权限，`::= { system 3 }` 把对象挂到命名树的具体位置上。厂商在 `enterprises` 分支下用同样语法定义私有对象，扩展自己的设备能力。

## SNMP版本与安全

### 版本演进

| 特性 | SNMPv1 | SNMPv2c | SNMPv3 |
|------|--------|---------|--------|
| 标准化年份 | 1990 | 1996 | 1998（2002 成熟） |
| 安全机制 | 团体名 | 团体名 | USM + VACM |
| 认证 | 明文团体名 | 明文团体名 | HMAC-MD5 / HMAC-SHA |
| 加密 | 无 | 无 | DES / AES |
| 批量获取 | 无 | GetBulk | GetBulk |
| 错误码 | 基本 | 增强 | 增强 |

- **v1**：最初版本，用明文"团体名 (community string)"做访问控制——相当于一个口令，分只读 (`public`) 和读写 (`private`)。无加密、无真正认证。
- **v2c**：在 v1 基础上增加 GetBulk、InformRequest 和更丰富的错误码，安全模型仍沿用团体名（"c" 即 community-based）。
- **v3**：引入真正的安全框架，提供认证、加密和细粒度访问控制。

注：团体名以明文随 UDP 传输，极易被窃听，因此 v1/v2c 不应在不受信网络上承载敏感操作。

### SNMPv3 的安全机制

SNMPv3 不改变 PDU 语义，而是增加了两个安全模块：

- **USM (User-based Security Model)**：基于"用户"提供三项安全服务——
  - **认证**：用 HMAC-MD5 或 HMAC-SHA 校验报文来源和完整性。
  - **加密**：用 DES/AES 加密 PDU，防窃听。
  - **时效性**：防止报文被重放。
- **VACM (View-based Access Control Model)**：基于"视图"控制某用户能访问 MIB 树的哪些子树（只读/读写）。

USM 提供三种安全级别：

| 安全级别 | 认证 | 加密 |
|----------|------|------|
| noAuthNoPriv | 否 | 否 |
| authNoPriv | 是 | 否 |
| authPriv | 是 | 是 |

## 本章小结

- **网络管理框架**三要素：管理服务器、被管设备（含代理与 MIB）、管理协议 SNMP。两种交互模式：管理服务器轮询，代理 Trap 主动上报。
- **SNMP** 运行在 UDP 上（代理 161、服务器 162）。核心 PDU 有 Get/GetNext/GetBulk/Set/Response/Trap/Inform。GetNext 用于遍历，Trap 用于主动告警。
- **MIB** 是被管对象集合，**SMI** 是描述 MIB 的语言；每个对象用全局 OID 命名，标准对象定义在 MIB-II（`1.3.6.1.2.1`）下。
- **版本**：v1/v2c 用明文团体名、无加密；v3 通过 USM（认证+加密）和 VACM（访问控制）提供安全。

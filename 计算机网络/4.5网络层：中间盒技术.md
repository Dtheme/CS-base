## 中间盒概述

> **中间盒 (Middlebox)**
>
> 位于源与目的之间的转发路径上、对分组执行**转发以外**功能的设备。它不只是按目的地址移动分组，而是检查、修改或控制分组，常工作在网络层之上（传输层、应用层）。

### 中间盒与端到端原则

互联网早期的设计哲学是**端到端原则 (end-to-end principle)**：把复杂功能放在端系统，网络核心只做尽量简单的"尽力而为"分组转发。中间盒在网络内部插入状态和处理逻辑，正是对这一原则的偏离。

> 易混：**端到端原则 vs 中间盒**。
> 端到端原则主张"功能放端系统、核心保持简单"；中间盒则把安全、地址复用、流量管理等功能下沉到网络内部。两者并非对错之分，而是简单性与可控性之间的权衡：NAT 缓解了地址耗尽，防火墙提供了集中安全，但也带来了部署复杂、破坏端到端透明性（如 NAT 阻碍 P2P、防火墙影响新协议）等代价。

| 特征 | 传统路由器/交换机 | 中间盒 |
|------|------------------|--------|
| 决策依据 | 目的地址 | 多字段 / 分组内容 |
| 状态 | 无连接状态 | 常维护连接/会话状态 |
| 处理层次 | 网络层 | 可达传输层、应用层 |
| 是否改包 | 一般不改 | 常重写头部甚至载荷 |
| 对端到端透明性 | 透明 | 可能破坏（如 NAT、代理） |

### 中间盒分类

按主要功能可分为四类：

```
中间盒
├─ 安全类    防火墙、IDS/IPS、WAF、安全网关
├─ 性能类    负载均衡器、缓存/代理、流量整形、CDN 边缘
├─ 网络功能  NAT、DHCP、DNS、VPN 网关
└─ 监控类    DPI、流量分析、网络审计
```

按相对数据通路的部署方式可分为：**串联（in-line，流量必经）** 与 **旁路（out-of-band，镜像分析）**，详见[第 6 节](#中间盒部署形态)。

## 防火墙技术

> **防火墙 (Firewall)**
>
> 部署在受保护网络与外部网络之间的中间盒，依据安全策略允许或阻止分组通过。按对分组的检查深度，分为无状态包过滤、有状态包过滤、应用网关三类。

防火墙的典型位置是内网与外网的边界（边缘路由器旁或集成在其上），所有进出流量都要经过它：

```
        内网                防火墙               外网/Internet
   ┌──────────┐         ┌──────────┐         ┌──────────┐
   │ 受保护主机│ ◄─────► │ 安全策略 │ ◄─────► │  外部主机 │
   └──────────┘         │  规则匹配 │         └──────────┘
                        └──────────┘
                     允许 / 拒绝 / 日志
```

### 无状态包过滤 (Stateless Packet Filter)

逐个分组独立判断，**不记忆此前分组**，仅根据头部字段匹配访问控制规则。常用判据是 IP 头与传输层头的若干字段（"五元组"）：

```
源IP  目的IP  源端口  目的端口  协议
  ↓
按规则表自上而下匹配，命中即执行动作（默认拒绝兜底）：
  allow tcp 192.168.1.0/24 -> any : 80
  deny  ip  any            -> any        ← 默认拒绝
```

特点：实现简单、转发快、对设备资源要求低；但因为不看连接状态，难以表达"只允许内部主动发起的连接的回包"这类规则，安全粒度粗。

注：教材给出的典型例子——只允许内网用户访问外部 Web（出向 80/443）、拒绝外部主动发起的连接，用无状态规则表达时需要小心放行 TCP 回包，这正是有状态过滤要解决的问题。

### 有状态包过滤 (Stateful Packet Filter)

在包过滤基础上**维护一张连接状态表**，跟踪每条 TCP/UDP 连接的状态，只放行属于已建立连接的分组：

```
       内网主机发起连接
            │
            ▼
   ┌─────────────────────────────────────────┐
   │ 连接状态表                                │
   │ 192.168.1.100:12345 ↔ 80.80.80.80:80     │
   │ 状态: ESTABLISHED   超时: 7200s          │
   └─────────────────────────────────────────┘
            │
   回包到达 → 查表命中已建立连接 → 自动放行
   无对应表项的外部主动连接 → 拒绝
```

相比无状态过滤，它能根据连接方向与状态自动放行回程流量、识别异常分组（如不属于任何连接的 ACK），安全性更强，代价是要维护状态、消耗内存。

易混：**无状态 vs 有状态包过滤**。无状态逐包独立判断、不看连接历史；有状态记录连接状态、只放行属于已有连接的分组。"出去的连接的回包能自动回来"是有状态过滤的核心能力。

### 应用网关 (Application Gateway)

也称代理防火墙，工作在应用层，针对特定应用（HTTP、FTP、SMTP 等）做代理：客户端连到网关，网关代为与服务器建立连接并转发，从而能解析并管控应用层内容。

```
客户端 ──► 应用网关(代理) ──► 服务器
        在网关上做应用层检查：
        身份认证、URL/命令过滤、内容审查
```

特点：能在应用层施加细粒度策略（如只允许特定用户经 Telnet 外联），但需为每种应用单独实现、处理开销大、对客户端不透明。

> 补充：**下一代防火墙 (NGFW)** 是工程概念而非教材内容，指把有状态过滤与应用识别、入侵防护（IPS）、用户身份等集成在一台设备上，能基于应用（而不只是端口）做访问控制。了解即可。

### 防火墙的局限

- **IP 欺骗**：包过滤按源 IP 判断，攻击者可伪造源地址。
- **加密流量**：无法检查 TLS 加密载荷的内容（应用网关需做 TLS 终结才能看到明文）。
- **规则复杂度**：规则一多就容易出现冗余、冲突、顺序错误。基本原则是"默认拒绝、具体规则在前、定期清理失效规则"。

## 网络地址转换设备

NAT 把多个内网私有地址复用为少量公网地址，是部署最广的中间盒。其工作机制、静态/动态/PAT 分类、NAT 表与穿透技术（STUN/TURN/UPnP）已在 [4.3 网络地址转换 NAT](4.3网络层：IP协议详解.md#网络地址转换nat) 完整讲解，此处仅从中间盒视角点出要点，避免重复。

**作为中间盒，NAT 做了什么**：在内网与外网边界改写 IP 地址与端口，并维护转换表把回程分组映射回内网主机。

```
内网                NAT                  外网
192.168.1.10:1234  ──►  改写源为  ──►  目的服务器
                        203.0.113.1:5678
  ◄── 改写目的回内网 ◄── 回包按 NAT 表映射 ──◄
```

**它对端到端原则的冲击**（教材重点）：

- 公网侧看到的是 NAT 的地址，内网主机的真实地址被隐藏，**外部无法主动发起到内网主机的连接**——这破坏了端到端可达性。
- 路由器本应只处理到网络层，NAT 却要读写传输层端口、重算校验和，**跨越了层次**。
- 端口字段被 NAT 用作主机标识，与"端口标识进程"的原意冲突。
- P2P、某些信令协议因此需要 STUN/TURN/UPnP 等**穿透技术**绕开 NAT（见 4.3）。

注：教材对 NAT 持批评态度，认为它违背端到端与分层原则，IPv6 才是地址耗尽的"正解"；但 NAT 因部署简单、隐藏内网而被广泛使用，二者长期并存。

## 负载均衡器

> **负载均衡器 (Load Balancer)**
>
> 把到达的请求/连接按某种策略分发到一组后端服务器的中间盒，用于横向扩展容量、提高可用性。它对客户端表现为单一虚拟服务地址。

### 分发算法

| 算法 | 依据 | 适用 |
|------|------|------|
| 轮询 Round Robin | 顺序循环 | 服务器能力相近 |
| 加权轮询 | 按权重比例 | 服务器能力不均 |
| 最少连接 | 当前活跃连接数最少 | 长连接、会话时长差异大 |
| 加权最少连接 | 连接数/权重 最小 | 能力不均 + 长连接 |
| IP 哈希 | hash(源IP) 选服务器 | 需要会话保持 |

```
轮询：    Req1→S1  Req2→S2  Req3→S3  Req4→S1 …
IP 哈希： hash(客户端IP) % N → 固定落到同一台（会话保持）
```

注：IP 哈希让同一客户端总落到同一后端，可在无共享会话存储时维持会话；代价是后端增减会导致大量重新映射。

### 健康检查与故障转移

负载均衡器需持续探测后端是否可用，常见三个层次：

- **TCP 检查**：能否完成三次握手。
- **HTTP 检查**：请求约定的 `/health` 路径，期望 200。
- **应用层检查**：执行一段真实业务逻辑（如一次数据库查询）确认服务真正可用。

连续多次失败即标记不健康、停止分发，并可启用备份后端；连续多次成功再标记恢复。

### 工作层次

- **四层 (L4)**：基于 IP+端口转发连接，不解析应用内容，性能高。
- **七层 (L7)**：解析 HTTP 等应用层信息，可按 URL、Host、Cookie 路由，并支持 TLS 终结、压缩等，功能强但开销大。

注：DNS 轮询和 CDN 的就近调度也能实现"分发"，但属于不同机制——DNS 轮询无法感知后端健康、受缓存影响；CDN 按地理位置选最近节点。它们与本节的串联式负载均衡器解决的问题层次不同。

## 深度包检测设备

> **深度包检测 (Deep Packet Inspection, DPI)**
>
> 不止检查 IP/传输层头部，而是深入到应用层**载荷内容**进行解析、识别与匹配的技术。是 IDS/IPS、流量管理、内容审计等功能的共同基础。

### 检测深度的层次

```
检查范围由浅入深：

浅层包检测  →  IP/传输层头部（源目的IP、端口、协议）
有状态检测  →  + 连接/会话状态
深度包检测  →  + 应用层载荷（协议解析、特征匹配）
深度流检测  →  + 跨多包的流行为/时序/统计特征
```

举例：对一个 HTTP 请求，浅层只看到"TCP 80 端口"，DPI 则能解析出请求方法、Host、User-Agent、Cookie 等，从而识别出"这是访问 www.example.com 的 GET 请求"。

### 应用识别方法

| 方法 | 原理 | 局限 |
|------|------|------|
| 端口号 | 知名端口映射协议（80→HTTP） | 端口可改、可隧道，易规避 |
| 载荷特征 | 匹配协议特征串（如 `SSH-2.0-`、`GET `） | 需维护特征库、加密后失效 |
| 行为特征 | 按流量模式判断（P2P 多对多、视频持续大下行） | 粗粒度、有误判 |
| 机器学习 | 用包长分布、到达间隔等特征分类 | 需训练、可解释性差 |

注：端口号识别是最早也最弱的方法，因为端口可任意修改、可用隧道伪装；现代 DPI 主要靠载荷特征与行为/统计特征。

### 典型应用

- **入侵检测/防护 (IDS/IPS)**：用特征匹配已知攻击（如在 HTTP 载荷中发现 `union select` 这类 SQL 注入特征），或用异常检测发现偏离基线的流量。IDS 只告警，IPS 可阻断。
- **数据泄露防护 (DLP)**：识别外发流量中的敏感数据（信用卡号、身份证号、机密关键词）并告警或阻断。
- **流量管理**：运营商/企业按应用类型做 QoS 与限速（如优先保障 VoIP/视频会议、压低 P2P）。

```
QoS 优先级示例（高 → 低）：
  VoIP / 视频会议  >  网页浏览  >  文件下载  >  P2P
```

### DPI 面临的挑战

- **加密**：HTTPS、VPN 流量的载荷不可见，DPI 只能看到元数据（SNI、流量大小/时序），深度内容检测失效。
- **性能**：高速链路上逐包解析载荷开销大，需专用硬件。
- **隐私**：深入检查用户内容涉及隐私与合规边界，需在监管与隐私之间权衡。

## 中间盒部署形态

中间盒相对数据通路的两种基本部署方式：

**串联 (in-line)**：流量必经设备，能实时阻断，但本身是单点与潜在瓶颈。防火墙、负载均衡器、IPS 一般串联。

```
Internet ──► 防火墙 ──► 负载均衡器 ──► 服务器群
```

**旁路 (out-of-band)**：交换机把流量**镜像**一份给设备分析，不在转发路径上，故不影响业务、设备故障也不影响通信，但**只能被动分析、无法实时阻断**。IDS、流量分析常旁路部署。

```
Internet ──► 交换机 ──► 服务器群
                │
            （流量镜像）
                ▼
             DPI / IDS（旁路分析）
```

实践中常**混合部署**：关键路径上串联安全设备（防火墙、WAF、负载均衡），同时旁路一套分析系统做监控审计。

注：是否串联取决于"是否需要实时阻断"。需要阻断（防火墙、IPS）就必须串联并接受单点风险；只做检测分析（IDS、审计）则旁路更稳妥。

此外，中间盒越来越多以**软件/虚拟化**形态运行——把防火墙、负载均衡等功能做成运行在通用服务器上的软件实例（网络功能虚拟化 NFV），便于按需扩缩与编排，代价是相对专用硬件的性能损失。SDN（[4.4](4.4网络层：通用转发与SDN.md)）的"匹配-动作"模型也能在交换机上实现防火墙、NAT、负载均衡等中间盒功能，是中间盒可编程化的方向。
